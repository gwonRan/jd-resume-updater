# Portfolio Case Builder

## 목적

사용자가 제공한 프로젝트 메모와 일부 자료를 기반으로 채용자가 읽기 쉬운 포트폴리오 케이스 스터디를 만든다. 포트폴리오는 결과물 전시가 아니라 문제 해결 방식 증명으로 다룬다.

## 기본 워크플로우

1. **입력 분류**
   - 목표 포지션/JD
   - 프로젝트 메모
   - 산출물 설명
   - 문제 상황과 목표
   - 사용자의 역할과 오너십
   - 스킬/도구
   - 협업/의사결정 맥락
   - 결과와 회고
   - 기밀 또는 비공개 정보

2. **프로젝트 증거 추출**
   프로젝트별로 문제, 목표, 역할, 해결 과정, 도구, 협업, 결과, 회고를 구조화한다.

   자세한 기준은 `references/portfolio_evidence_framework.md`를 사용한다.

   포트폴리오에 넣을 가치가 있는 프로젝트인지 먼저 판단해야 하면 `references/portfolio_fit_framework.md`를 사용한다.

3. **케이스 스터디 구성**
   각 프로젝트를 `문제 정의 → 목표 → 내 역할 → 해결 과정 → 사용한 스킬/도구 → 협업/의사결정 → 결과 → 회고 → JD 연결 역량` 구조로 만든다.

   구조 기준은 `references/portfolio_case_framework.md`를 사용한다.

4. **성과와 스킬 맥락화**
   단순 결과물 설명을 "왜, 어떻게, 무엇이 달라졌는가" 중심으로 바꾼다. 스킬/도구는 사용 맥락과 결과에 연결한다.

   프로젝트 유형별 구조가 필요하면 `references/project_type_templates.md`를 사용한다. 성과를 보여줄 시각화 아이디어가 필요하면 `references/visualization_guide.md`를 사용한다.

5. **품질 검토**
   채용자가 10초 안에 문제, 역할, 결과를 파악할 수 있는지 확인한다.

   기준은 `references/portfolio_quality_rubric.md`를 사용한다.

6. **후속 질문**
   문제 정의, 개인 기여, 성과 수치, 의사결정, 회고가 부족하면 질문한다.

   질문 패턴은 `references/followup_questions.md`를 사용한다.

## 기본 출력 순서

사용자가 출력 형식을 따로 지정하지 않으면 다음 순서를 따른다.

1. 입력 정보 요약
2. 확정 정보 / 확인 필요 / 부족 정보 / 민감 정보 분리
3. 포트폴리오 적합도 판단
4. 프로젝트 증거표
5. 케이스 스터디 목차
6. 프로젝트별 초안
7. 시각화/자료화 아이디어
8. JD와 연결되는 역량
9. 후속 질문

## 작성 원칙

- 사용자가 제공한 정보만 확정 사실로 사용한다.
- 포트폴리오를 예쁜 문서보다 문제 해결 구조로 본다.
- 결과 화면이나 산출물만 나열하지 않는다.
- 개인 기여와 팀 성과를 구분한다.
- 성과 수치를 임의로 만들지 않는다.
- 실패, 제약, 회고는 숨기지 않고 신뢰도를 높이는 재료로 사용한다.
- 내부 정보와 민감 수치는 일반화한다.

## 신뢰도 라벨

- **확정**: 제공된 자료에서 직접 확인됨
- **확인 필요**: 가능성은 있으나 사용자 확인이 필요함
- **부족**: 케이스 스터디 품질을 위해 추가 정보가 필요함
- **민감**: 외부 공개 전 익명화 또는 일반화가 필요함

## 출력 모드

- 포트폴리오 목차
- 프로젝트 증거표
- 프로젝트별 케이스 스터디
- 문제 정의/해결 과정/결과/회고 초안
- 스킬/도구 맥락표
- 성과 시각화 아이디어
- 포트폴리오 첫 페이지 요약
- JD와 연결되는 역량 매핑
- 후속 질문 리스트

## 입력 템플릿

사용자가 어떤 자료를 제공해야 하는지 묻는다면 `assets/input_template_ko.md`를 제공한다.
