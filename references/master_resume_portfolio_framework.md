# 마스터 이력서/포트폴리오 프레임워크

마스터 이력서, 마스터 포트폴리오, 성과 언어 변환, 스킬/도구 맥락화, 기간/기한 추출이 필요한 요청에서 사용한다.

## 핵심 원칙

- 모든 경험은 먼저 마스터 재료로 모은다.
- 제출본은 JD에 맞춰 마스터에서 관련 경험을 고르고, 관련성이 낮은 경험은 삭제하거나 축소한다.
- 단순 활동은 성과 언어로 바꾼다.
- 스킬과 도구는 이름만 나열하지 않고 사용 맥락과 결과에 연결한다.
- 기간, 기한, 마감, 지연, 단축, 운영 기간은 프로젝트 난이도와 임팩트를 보여주는 근거로 추출한다.
- 작은 경험도 작게 만들지 않되, 하지 않은 일을 했다고 쓰지 않는다.

## 마스터 이력서/포트폴리오 만들기

마스터 문서는 제출용 완성본이 아니라 재료 저장소다. 가능한 많은 경험을 구조화해 두고, JD별 제출본을 만들 때 필요한 것만 꺼내 쓴다.

### 수집 항목

- 회사/조직
- 직무/역할
- 기간
- 프로젝트명 또는 경험명
- 문제 상황
- 목표
- 내 역할과 오너십 범위
- 구체적 행동
- 사용한 스킬, 도구, 방법론
- 협업 대상
- 산출물
- 정량 성과
- 정성 성과
- 배운 점/회고
- 공개 가능 여부
- 관련 JD 키워드

### JD별 제출본 만들기

1. JD에서 필수 역량, 우대 역량, 반복 키워드, 직무 레벨 신호를 뽑는다.
2. 마스터 재료에서 JD와 직접 연결되는 프로젝트를 고른다.
3. 관련성이 낮은 경험은 과감히 줄인다.
4. JD 키워드는 한 번 이상 자연스럽게 반영하되, 같은 키워드를 과하게 반복하지 않는다.
5. 비슷한 직무군은 하나의 버전으로 묶고, 다른 직무군은 별도 버전으로 만든다.

## 성과 언어로 바꾸기

사용자가 단순 활동을 제공하면 다음 순서로 변환한다.

1. 원래 활동을 확인한다.
2. 그 활동이 해결하려던 문제나 목표를 찾는다.
3. 사용자가 한 구체적 행동을 찾는다.
4. 사용한 방법, 도구, 협업 방식을 연결한다.
5. 결과를 정량 또는 정성 성과로 표현한다.

### 변환 공식

`원래 활동 + 사용한 방법/도구 + 변화한 결과`

또는:

`문제 → 해결 방식 → 결과`

### 예시

약한 표현:

`Jira를 사용해 백로그 관리 및 일정 조율`

성과 언어:

`Jira 기반 스프린트 계획을 세우고 개발자별 업무 진행률을 트래킹해 일정 지연 리스크를 조기 발견`

성과 수치가 확인된 경우:

`Jira 기반 스프린트 계획과 진행률 트래킹을 도입해 일정 지연을 20% 감소`

## 스킬/도구 넣기

스킬과 도구는 두 방식으로 정리한다.

### 목록형

이력서의 핵심 역량 또는 Skills 섹션에 사용한다.

- 프로젝트 관리: Jira, Asana, Trello
- 협업/문서화: Confluence, Notion, Slack
- 데이터/분석: SQL, Python, GA4
- 기술 이해: AWS, Docker, React, Django

### 맥락형

프로젝트 bullet 또는 포트폴리오 케이스 스터디에 사용한다.

`[도구/스킬]을 활용해 [업무/프로세스]를 수행하고, [결과/효과]를 만들었다.`

예:

`Confluence와 Notion을 활용해 요구사항, 의사결정, 변경 이력을 문서화하고 개발/디자인/QA 간 커뮤니케이션 비용을 낮춤`

## 기간/기한 넣기

시간 정보는 프로젝트의 난이도, 속도, 운영 경험, 일정 관리 능력을 보여준다.

### 추출할 시간 정보

- 프로젝트 시작/종료 시점
- 재직기간
- 출시일 또는 마감일
- 원래 목표 일정
- 일정 지연 기간
- 일정 단축 폭
- 반복 운영 기간
- 개선 전/후 비교 기간
- 실험 기간

### 문장 반영 방식

- `N개월간 [프로젝트/운영]을 담당`
- `[마감일/출시일]에 맞춰 [산출물]을 완료`
- `[지연 기간] 지연된 프로젝트에서 [행동]을 통해 일정 리스크를 줄임`
- `[기간] 동안 [지표]를 모니터링하고 [개선]을 실행`

## 포트폴리오 케이스 스터디 구조

포트폴리오는 결과물 전시가 아니라 문제 해결 방식 증명이다.

### 권장 구조

1. 프로젝트 요약
2. 문제 정의
3. 목표와 성공 기준
4. 내 역할과 오너십
5. 해결 과정
6. 사용한 스킬/도구
7. 협업과 의사결정
8. 결과
9. 회고와 다음 개선점
10. JD와 연결되는 역량

### 좋은 포트폴리오의 기준

- 왜 이 프로젝트를 했는지 보인다.
- 어떤 문제를 어떻게 정의했는지 보인다.
- 사용자가 어떤 판단과 행동을 했는지 보인다.
- 결과가 수치나 구체적 변화로 보인다.
- 실패, 제약, 회고가 있으면 더 신뢰도가 높아진다.
- 디자인보다 읽는 사람이 10초 안에 핵심 성과를 파악할 수 있는 구조가 중요하다.

## 후속 질문

정보가 부족하면 다음 질문 중 필요한 것만 묻는다.

- 이 프로젝트는 언제부터 언제까지 진행됐나요?
- 원래 목표 일정이나 마감일이 있었나요?
- 일정 지연, 단축, 출시일 준수 같은 시간 관련 성과가 있나요?
- 어떤 스킬/도구를 사용했고, 그 도구로 무엇을 가능하게 했나요?
- 성과를 전/후, 목표 대비, 증가/감소율로 표현할 수 있나요?
- 이 경험은 마스터 이력서에 남길 핵심 경험인가요, 특정 JD에서만 쓸 보조 경험인가요?
- 포트폴리오 케이스 스터디로 확장할 만큼 문제 정의와 해결 과정이 충분한가요?
